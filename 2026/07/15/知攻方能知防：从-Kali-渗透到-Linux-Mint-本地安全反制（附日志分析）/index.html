知攻方能知防：从 Kali 渗透到 Linux Mint 本地安全反制（附日志分析）

引言：为什么我们要学习“双向攻防”？

在网络安全行业中，“不懂防守的黑客成不了大师，不懂攻击的安全官无法保护企业。”

今天，我亲自在一台作为攻击机的 Kali Linux 和一台作为靶机的 Linux Mint 主机之间，完成了一次从端口扫描、SSH 爆破登录、本地提权，到防守方日志审计、网络连接排查、最终精准 Kill 进程反制的闭环实战。

本文将以“红蓝对抗”的双视角，带大家沉浸式还原整场博弈。

🟥 红队视角：步步蚕食，夺取 Root 权限

1. 探路：网络扫描与穿透

首先，利用 Nmap 探测目标 Linux Mint 主机的开放端口，发现其 22 端口（SSH 服务）处于开启状态：

nmap -Pn <目标IP>

2. 渗透：建立 SSH 会话

在 Metasploit（MSF）框架下，利用已获取的凭据成功建立 SSH 连接：

msf > use auxiliary/scanner/ssh/ssh_login

3. 登顶：本地提权

登录为普通用户 panda 后，利用 sudo -l 查看特权，随后执行：
sudo su

4. 权限的利用

在 Root 权限下，肆无忌惮地读取机密文件 /etc/shadow，并企图维持长期的后台控制。

cat /etc/shadow

【蓝队实战】记一次 SSH 暴力破解及权限提升事件的应急响应与溯源分析

0x01 事件背景

在日常的安全运营监控中，发现一台生产环境的 Linux Mint 主机（IP: 192.168.249.128）出现异常的外部连接，并伴随有疑似提权操作。

作为应急响应人员（IR），我立刻切入该主机，在不重启系统、不中断正常业务的前提下，展开了法医级的“溯源排查”与“现场网络阻断”。

0x02 应急排查：还原黑客攻击链路

黑客以为自己神不知鬼不觉，但在 Linux 主机的底层日志里，其实早已留下了无法抹除的指纹。

1. 入侵源头审计：安全日志分析（Auth Log）

首先，蓝队必须找到黑客是如何进来的。我们直接调取系统的安全日志 auth.log：

过滤 SSH 相关安全日志

sudo grep “sshd” /var/log/auth.log | tail -n 20

过滤 Sudo 提权日志

sudo grep “sudo” /var/log/auth.log | tail -n 15

普通权限查询（错误示范）

ss -antp | grep 22
由于没有管理员权限，系统为了安全会隐藏关键信息。输出结果里只会显示连接，而最后一列的 PID（进程ID）会被隐藏（留空），导致我们无法锁定内鬼进程。

正确排查命令

sudo ss -antp | grep 22

区分连接状态（ESTAB vs LISTEN）

  1. LISTEN 状态：代表本机的 SSH 守护进程正在监听（如 0.0.0.0:22）。绝对不能杀，否则整个 SSH 服务将瘫痪，后续管理员也无法登录。
  2. ESTAB 状态：代表已经建立的活动连接。（establish）
  3. 最后：在输出的 ESTAB 这一行中，我们成功抓到了指向 192.168.249.129:39929（黑客 IP）的活动连接，并在最后一列精准定位到了黑客会话的进程 ID（PID）

强行杀死属于黑客连接的特定进程

sudo kill -9 3161
