浅浅的了解一下越权是什么东东(狗头保命)？

前言

越权的简单理解：
干了不该干的事情 或者看了不该看的数据

指的是应用程序在验证了用户的身份（知道你是谁）之后
没有严格检查你的权限，导致普通用户执行了只有特定人（如管理员或其他用户）才能执行的操作。

越权的两大核心类型

  1. 水平越权（同级越权）
  • 概念：发生在权限级别相同的用户之间。你可以访问或修改同等级别其他用户的数据
  • 常见场景：

    在查询个人订单时，网址是 …/order?id=1001。
    如果你把 1001 改成 1002，系统没有校验 1002 是不是你的，直接把别人的订单详情、地址、手机号显示出来了。

  1. 垂直越权（跨级越权）
  • 概念：发生在权限级别不同的用户之间。低权限的用户成功执行了高权限（如管理员）才能做的事。
  • 常见场景：

    普通用户登录后的后台管理页面地址是 …/admin/delete_user。
    普通用户登录后的后台管理页面地址是 …/admin/delete_user。

越权漏洞的产生

本质

系统在处理用户请求时，只验证了“你是谁”（身份认证，Authentication），而忽略或漏掉了“你能干什么”（权限授权，Authorization）。

前端的原因

  1. 直接暴露对象标识符（IDOR - Insecure Direct Object References）
  • 系统直接把数据库的主键（如用户 ID user_id=101、订单 ID order_id=5002）放在请求参数或 URL 中。当用户把 101 改成 102 时，服务器没有校验当前登录用户是否拥有 102 号数据的读取权限。
  1. 前端隐藏敏感按钮
  • 攻击者不需要通过点击界面上的按钮来发请求。他们可以通过抓包工具，直接向服务器发送这个“敏感按钮”背后对应的 API 接口指令（例如 POST /api/user/delete）。如果后端没有做权限检查，操作依然会成功执行——这就是垂直越权

后端的原因

  1. 未确认用户已登录
  • 程序在收到请求后，确实检查了用户的 Cookie 或 Token（确认了合法身份），但没有查询数据库来核实该数据是否属于该用户。
  • 需要进行第二次校验来与第一次的user_id进行对比，看是否对该具体资源拥有权限。

访问控制机制设计不严密

  1. 随着业务越来越复杂，系统的权限层级也会增多
  • 硬编码漏洞：权限控制分散写在各个接口的代码里，容易漏写。比如开发了 100 个后台接口，其中 99 个都加了管理员权限检查，但有 1 个接口忘记加，就会产生垂直越权。
  • 默认配置不安全：路由或中间件配置错误，例如误将管理后台的接口路径（如 /api/admin/*）配置成了公开访问或仅需普通登录访问。
  1. 不可靠的身份验证
  • 默认配置不安全：路由或中间件配置错误，例如误将管理后台的接口路径（如 /api/admin/*）配置成了公开访问或仅需普通登录访问。
  • 前端传参决定身份：某些不规范的接口，让前端直接提交 is_admin=true 或 role=admin 参数，服务器未经校验就直接信任了客户端传过来的身份字段。
